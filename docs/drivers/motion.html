Zumly + Motion

Drive Zumly's zoom-in, zoom-out, and lateral transitions through Motion (motion.dev) — the modern successor to Popmotion and the engine behind Framer Motion. Spring curves, hardware-accelerated keyframes, and matrix-faithful intermediate frames in one option.

Get started → View on GitHub

Why use the Motion driver?

Motion (motion.dev) is a tiny, modern animation library that compiles down to native WAAPI calls where it can. Pick it for Zumly when you want:

  • Spring physics as a first-class ease — natural, settled motion that feels especially right for "snap into focus" zoom interactions.
  • Hardware-accelerated keyframes via the underlying WAAPI bridge, without having to write keyframe arrays yourself.
  • Matrix-interpolated intermediate frames — like the Anime.js driver, the Motion adapter interpolates the computed CSS matrix so transform-origin shifts don't cause visual pops.
  • Tiny payload — Motion's mini build is one of the smallest animation libraries you can drop on a page.

Install

The Motion adapter is bundled inside zumly. Add Motion as a global:

npm install zumly
<script src="https://cdn.jsdelivr.net/npm/motion@11/dist/motion.min.js"></script>
Bundler users: if you import Motion from npm, assign it to globalThis.Motion = Motion (or globalThis.motion) before constructing Zumly. The adapter looks for either name.

Quick start

import Zumly from 'zumly'
import 'zumly/css'

const app = new Zumly({
  mount: '.canvas',
  initialView: 'home',
  views: { home, room, device },
  transitions: {
    driver: 'motion',
    duration: '600ms',
    ease: 'ease-in-out',
  },
})

await app.init()

That's it. Zumly resolves 'motion' to the bundled Motion adapter and runs each transition through Motion's animate() primitive.

Working example: spring-curve zoom

Spring curves are where Motion really earns its place in a Zumly app. Here a home-control UI zooms into a room, then a device, with a slightly underdamped spring that lets the view "settle" rather than slam to a stop. 

const app = new Zumly({
  mount: '.canvas',
  initialView: 'home',
  views: { home, room, device },
  transitions: {
    driver: 'motion',
    duration: '650ms',
    ease: 'spring(1, 100, 12, 0)',  // mass, stiffness, damping, velocity
    effects: ['blur(3px) brightness(0.7)', 'blur(7px) brightness(0.4)'],
  },
})

await app.init()

Any string Motion accepts as an easing — including its named curves ('easeOut', 'circOut', etc.) and spring shorthand — is passed through. Standard CSS easing strings are also accepted and mapped to a sensible default.

What the adapter actually does

  • Calls showViews() on current, previous, and grandparent views.
  • Reads start and end matrices via readComputedMatrix() for each view.
  • Drives a Motion animate({ progress: 0 }, { progress: 1 }, ...) tween and, on each onUpdate, applies the interpolated matrix as the view's transform.
  • On finish, calls the canonical applyZoomInEndState() / applyZoomOutPreviousState() / applyZoomOutLastState() helpers and then onComplete().

Lateral transitions are handed off to runLateralInstant() — Zumly's lateral slide is geometric and doesn't benefit from a curve.

Direct import (advanced)

import Zumly, { motionTransition } from 'zumly'

const app = new Zumly({
  mount: '.canvas',
  initialView: 'home',
  views: { home, detail },
  transitions: {
    driver: motionTransition,
    duration: '500ms',
  },
})

Wrapping the function lets you log every transition, compose with another animation, or swap drivers per transition type. The full spec shape is documented in the custom driver guide.

Common gotchas

  • Motion v11 only. The bundled adapter targets the current Motion (motion.dev) API. Older Popmotion or pre-11 Motion releases expose a different surface and won't work without a wrapper.
  • Spring strings vs duration. When you pass a spring ease, Motion ignores the explicit duration in favour of the spring's settling time. The adapter still uses your declared duration to size its safety timeout, so very long springs may need a higher duration value.
  • Matrix interpolation forces one extra layout read per transition. On modern devices this is invisible; on very low-end mobile, the css driver (which animates the transform string directly) is slightly cheaper.
