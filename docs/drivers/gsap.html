Zumly + GSAP

Drive Zumly's zoom-in, zoom-out, and lateral transitions with the GSAP timeline engine. The GSAP driver is bundled with Zumly — no adapter, no glue code, just a one-line constructor option and the eases you already know.

Get started → View on GitHub

Why use the GSAP driver?

Zumly ships with several transition drivers — css (default), waapi, none, and adapters for gsap, anime, and motion. The driver is the layer that decides how a view moves; the engine decides what moves where.

Pick GSAP when you want:

  • GSAP's ease catalog — power2.inOut, back.out, elastic, expo and the rest, mapped automatically from your Zumly ease string.
  • Timeline-based stagger — current view, previous view, and grandparent animate on a single GSAP timeline so they stay in sync even when you tune stagger.
  • Existing GSAP investment — if your app already loads GSAP for other animations, reusing it for Zumly transitions costs you zero extra bytes.

Install

The driver is part of zumly. You only need to make GSAP available as a global.

npm install zumly

Then load GSAP from a CDN before Zumly initialises:

<script src="https://cdnjs.cloudflare.com/ajax/libs/gsap/3.12.5/gsap.min.js"></script>
Why a global? The bundled adapter reads globalThis.gsap so Zumly stays zero-dependency at the package level. If GSAP isn't on the page, the driver warns once and falls through to onComplete() — your views still navigate, just without GSAP's curve.

Quick start

Pass 'gsap' as the driver name. That's the entire integration. 

import Zumly from 'zumly'
import 'zumly/css'

const app = new Zumly({
  mount: '.canvas',
  initialView: 'home',
  views: { home, dashboard, detail },
  transitions: {
    driver: 'gsap',
    duration: '600ms',
    ease: 'ease-in-out',
  },
})

await app.init()

Zumly resolves 'gsap' to the bundled adapter, normalises your ease string to a GSAP curve ('ease-in-out' → 'power2.inOut'), and runs the transition on a GSAP timeline.

Working example: tuned curve and stagger

GSAP shines when you push beyond linear interpolation. This example uses an elastic ease for zoom-out and a custom stagger so the parent view starts moving slightly after the current view. 

const app = new Zumly({
  mount: '.canvas',
  initialView: 'overview',
  views: { overview, region, sensor },
  transitions: {
    driver: 'gsap',
    duration: '700ms',
    ease: 'back.out(1.4)',     // GSAP-native ease string
    stagger: 80,                // ms — applied between views[0], views[1], views[2]
    effects: ['blur(4px) brightness(0.6)', 'blur(10px) brightness(0.3)']
  },
})

await app.init()

Any GSAP ease that gsap.to() accepts works directly. Numeric strings like 'power3.inOut', 'expo.out', or 'elastic.out(1, 0.3)' are passed through without translation.

What the adapter actually does

Internally the GSAP driver builds one timeline per transition. For a zoom-in:

  • It calls showViews() on current, previous, and (if depth > 2) grandparent.
  • It snapshots the engine-computed start transforms via setTransform().
  • It animates each view's transform from its backwardState to its forwardState with the resolved duration and ease.
  • On onComplete it calls applyZoomInEndState() for each view so the DOM ends in the canonical "current view" state Zumly expects.

Lateral transitions are handed off to runLateralInstant() — the same instant helper used by minimal custom drivers — because GSAP doesn't add value to Zumly's slide-and-snap lateral motion.

Direct import (advanced)

If you want to wrap or extend the driver — for example, to log every timeline or to compose it with another animation — you can import the function and pass it directly: 

import Zumly, { gsapTransition } from 'zumly'

const app = new Zumly({
  mount: '.canvas',
  initialView: 'home',
  views: { home, detail },
  transitions: {
    driver: (spec, onComplete) => {
      console.log('zumly transition', spec.type, spec.currentStage)
      gsapTransition(spec, onComplete)
    },
    duration: '500ms',
  },
})

Passing a function is the official way to extend any built-in driver — see the custom driver guide for the full spec shape.

Common gotchas

  • Load GSAP before Zumly initialises. The driver checks globalThis.gsap at construction time. ESM-only setups should still drop a script tag, or assign globalThis.gsap = gsap after importing.
  • Don't tween the same views from your own GSAP code mid-transition. Zumly owns the transform property during a zoom; concurrent tweens will fight the timeline.
  • Stagger is in milliseconds. Zumly's stagger option is ms-based; the driver converts it internally to seconds for GSAP's timeline API.
  • Lateral transitions are intentionally instant. If you want lateral motion, use the waapi driver (which has a built-in lateral slide) or author a custom driver that handles spec.type === 'lateral' yourself.
