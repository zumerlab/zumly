Depth navigation

Depth navigation is the zoom-in / zoom-out axis of a ZUI — the way you drill from a high-level view into one of its children. In Zumly, every depth move is a single primitive: zoomTo(viewName) takes you in, back() takes you out.

The zoom hierarchy

Every Zumly app has, at any given moment, a stack of views ordered by depth. At depth 1 you're looking at the initial view. Click a trigger and you're at depth 2 — the triggered view fills the canvas, the initial view is still there as the parent. Click again and you're at depth 3, and so on.

The engine keeps three views alive on the canvas at the deepest practical depth:

  • Current view — the one you're focused on, filling the viewport.
  • Previous view — the parent, scaled down and tucked behind / around the current view. Visible enough to give context, faded enough to stay out of the way.
  • Last view — the grandparent, kept on the canvas so a zoom-out animation has somewhere to land. Only present at depth > 2.

Anything older than the grandparent is removed from the DOM. This isn't a limitation — it's a deliberate choice that keeps memory bounded no matter how deep your zoom stack gets, while still preserving the spatial illusion that "everything is still there."

How a zoom-in actually works

When the user clicks an element with class="zoom-me" and data-to="targetView", Zumly:

  • Reads the trigger element's bounding rectangle relative to the current view.
  • Computes the transform that would scale the current view so the trigger fills the viewport.
  • Renders the target view as a fresh DOM subtree, sized to match where the trigger used to be.
  • Hands off to the active transition driver, which animates the current view up to fill the viewport while the parent slides out and back to make room.
  • On completion, marks the new view as .is-current-view and updates the depth tracker.

Crucially, the trigger element's screen position is the start of the animation — that's why the destination feels physically connected to where the user clicked. There's no jarring page transition; the spatial relationship is preserved.

How a zoom-out works

Zoom-out is the reverse, with one important detail: the engine has to recompute the layout for the previous view because the canvas may have been resized, the parent's layout may have changed, or the user may have rotated their device since the zoom-in happened. Zumly batches all those reads in a single layout pass before handing off to the driver — see the API reference for the full story.

After the animation, the current view is removed from the DOM, the previous view becomes the new current, and the grandparent (if any) is promoted to "previous."

Why depth, not URLs?

A traditional flat-page app uses URLs as the navigation primitive. URLs are great for sharing, indexing, and remembering — but they're terrible at expressing spatial relationships. The URL /dashboard/region/sensor-42 tells you the path through the hierarchy, but it doesn't tell you that "sensor 42" is the thing in the bottom-right corner of "region 7" which is the green tile in the dashboard.

Depth navigation expresses that spatial relationship as the navigation primitive itself. The trade-off is that states are not addressable — there's no URL for "sensor 42 is currently focused" because the focused state depends on which trigger element you came from. That's a feature, not a bug: it means the back button always does the right thing (zoom out, no matter how you got here), and you can't accidentally deep-link into a state where the parent context is missing.

For SEO: if you want crawlable, shareable URLs for parts of a Zumly app, build them as static HTML pages alongside the SPA — exactly the pattern this docs site uses. The static pages link to the SPA home, not into a specific zoom state.

Three depth limit isn't a limit

Zumly only keeps three views on the canvas at once, but your zoom stack can be arbitrarily deep. The engine treats older ancestors as "off-canvas but logically still in the stack" — back() walks the full stack, the engine just doesn't render ancestors the user can't see.

In practice, real ZUI apps rarely go past depth 4 or 5 anyway. If you find yourself building a 10-level drill-down, that's usually a hint that one of those levels should be a lateral move instead — same depth, different sibling.
