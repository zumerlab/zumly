Focus-driven UI

A focus-driven UI is one where the central design question is "what is the user looking at right now" — and everything else (controls, siblings, ancestors) recedes until they're needed. ZUIs are the most direct way to build one.

The principle

Most interfaces compete for attention. A flat dashboard shows you fifteen widgets at once — three of them blinking, two with red badges, and a sidebar full of menu items you'll never click. The user's brain has to filter all of this every time they look at the screen.

A focus-driven UI inverts the default. The thing the user is looking at is the entire screen. Siblings exist but they're peripheral — visible enough to remind the user that they're there, faded enough that they don't compete. Parents and ancestors are even more recessed: they're context, not content.

Three rules of focus-driven design

1. One focus at a time

Whatever you've zoomed into is the screen. Don't try to keep a "main panel" and a "side panel" co-equal. If two things deserve equal focus, they're either siblings (use lateral navigation) or they belong in a single composite view.

2. Context belongs at the edges

The parent view should remain visible — usually scaled down and behind the focused view — so the user always knows where they are in the hierarchy. Don't hide it. Don't replace it with a breadcrumb trail. The whole point of a ZUI is that the parent is still there, physically.

3. Transitions are part of the UI, not decoration

In a focus-driven UI, the zoom animation isn't eye candy — it's the way the user understands "I'm now inside this thing." Skipping the transition (or making it too fast to perceive) breaks the spatial illusion and the user gets lost. The transition driver is a load-bearing part of the design, not a polish layer.

Where focus-driven UIs are the right answer

  • Kiosks and signage. Public terminals where the user has zero training, low patience, and one task at a time. Focus-driven UIs are forgiving because there's only ever one "right next move."
  • Vehicle and equipment dashboards. Driver/operator UIs where attention is precious and competing widgets are dangerous. Zumly's production line showcase is exactly this.
  • Home and IoT control. Rooms full of devices, each device with its own settings. The natural mental model is "show me the room, then the device, then its controls" — see the Home Assistant showcase.
  • Mission control and ops centres. Dense data with deep drill-down. See the mission control showcase.
  • Authoring tools and configurators. Where the user wants to focus on one piece, edit it, and pop back to the whole — without losing context.

What focus-driven UI is not

Focus-driven design isn't minimalism. It doesn't mean "show less stuff." It means "make attention explicit." A focused view can be as dense as it wants — what matters is that the user is unambiguously inside that view, and the rest of the app has politely stepped aside.

It's also not the same as full-screen mode. Going full-screen drops the context entirely; focus-driven UIs preserve context at the edges of the viewport. That's the difference between "I am now in this thing" and "the rest of the world has disappeared."

Designing the trigger

In a focus-driven ZUI, the trigger element becomes the next view. That changes how you design clickable elements: they're not buttons that send the user somewhere else, they're previews of where the user is about to go. A card you can zoom into should look like a small version of the view it expands to — same colours, same layout, same content density (just compressed).

Concretely:

  • Cards in the parent view should mirror the layout of the child view they expand to.
  • Avoid placing the trigger in a way that doesn't physically connect to the destination — random "Open" buttons in a table row break the spatial illusion.
  • Make the trigger element the right size for the user to read it in the parent context. If the card is too small to scan, the user doesn't know what they're zooming into; if it's too big, the parent loses overview.
