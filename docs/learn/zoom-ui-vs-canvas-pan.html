Zoom UI vs canvas pan

Pannable canvases (Figma, Miro, Excalidraw, tldraw) and zooming user interfaces (Zumly, Pad++, Prezi) look superficially similar — both let you scale and move things — but they solve completely different problems. Knowing which you need is the most important architectural choice you'll make.

What is a ZUI? → Focus-driven UI

The one-line difference

A pannable canvas is a single, flat coordinate space that the user can navigate freely with pan + zoom. A ZUI is a discrete hierarchy of views that the user navigates by zooming into elements that become the next view.

In a canvas, position and zoom are continuous. In a ZUI, position and zoom are states — and the engine animates between them.

Side by side

Pannable canvas (Figma, Miro, tldraw)

  • Coordinate space: one continuous 2D plane.
  • Navigation: drag to pan, scroll/pinch to zoom, anywhere, any amount.
  • Hierarchy: implicit. Things are "near" or "inside" each other only by their xy coordinates and z-order — there's no formal parent/child navigation.
  • Back button: usually means "undo," not "previous view."
  • State addressability: often a viewport URL (e.g. Figma's ?node-id=42) that encodes pan + zoom + selection.
  • Best for: creative work, brainstorming, diagrams, design, anything where the user is producing or rearranging spatial content.

Zooming UI (Zumly, Pad++, Prezi, Deep Zoom)

  • Coordinate space: a tree of views. Each view is its own coordinate space; transitions between views are computed transforms.
  • Navigation: click a trigger to zoom into a child view; click "back" to zoom out.
  • Hierarchy: explicit. Every view has a parent (except the root). The engine knows the depth stack and walks it on back().
  • Back button: always means "zoom out one level."
  • State addressability: typically not addressable per state — a hash router collapses many depth states into one URL by design.
  • Best for: consuming hierarchical data, drilling into focus, kiosk/dashboard/control interfaces, anything where the user is reading and selecting rather than authoring.

The trigger is the giveaway

The clearest test: where does the next view come from?

In a canvas, when you "zoom in," you're scaling the entire current view up. The thing that ends up filling your screen is whatever happened to be in the centre of the viewport when you zoomed. The user is in control of the camera.

In a ZUI, you don't zoom the camera — you zoom into a specific element. That element becomes the next view. The trigger you clicked is the destination. The engine is in control of the geometry; you're in control of the choice.

Why it matters for the design

Canvases are great when the spatial layout is the content — moving sticky notes on a whiteboard, arranging shapes on a poster, sketching a diagram. The user's mental model is "I'm looking down at a table covered in stuff."

ZUIs are great when the spatial layout is the navigation — drilling from a dashboard into a chart into a row, or from a building into a room into a device. The user's mental model is "I'm walking through a structure."

Mixing the two in one interface usually fails. Canvas users expect to drag the world around; ZUI users expect to click things and have the world come to them. If you try to support both gestures on the same surface, both groups of users will be confused about which one they're in.

"But Prezi has both"

Prezi looks like a canvas because it lets you arrange slides on a 2D plane. But the navigation is ZUI: you don't pan to the next slide, you click a trigger (the "next" arrow or a step waypoint) and the engine zooms you into a specific coordinate. The canvas-like layout is for the author; the experience for the audience is pure ZUI.

That split is a useful pattern. You can build a canvas authoring tool for people who design ZUI flows, and the resulting flow can still be a strict ZUI for the end user. They're not the same product.

Where Zumly sits

Zumly is unambiguously a ZUI engine. There's no pan gesture, no continuous zoom, no viewport URL. Every navigation move is a click on a trigger element with an explicit destination, and the engine handles the geometry. That choice is what makes it small — the entire engine is roughly the size of a typical router — and what makes it composable with React, Vue, Svelte, Angular, or plain HTML views.

If you need a pannable canvas, Zumly isn't the right tool. If you need a focus-driven, hierarchical UI where the user reads and drills rather than authors and arranges, Zumly is exactly what it's for.
