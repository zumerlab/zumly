What is a ZUI?

A Zooming User Interface (ZUI) is an interface where navigation happens in space rather than across pages. Instead of swapping one screen for another, you zoom into the element you care about — and the rest of the world keeps existing around it.

The shortest possible definition

In a flat UI, the navigation primitive is "go to a different page." In a ZUI, the navigation primitive is "zoom into that thing right there." Whatever you click becomes the new viewport — visually, semantically, and as the new "you are here" of your app.

That single change does a lot of work. It tells the user where they came from (you can still see the parent receding behind the new view), it tells them what they zoomed into (the trigger element becomes the next view), and it gives them an obvious way back: zoom out.

A short history

The idea predates the web. In 1993 Ken Perlin and David Fox built Pad, a prototype of an "infinite zoomable surface" at NYU. Pad++ followed in 1994, then Jazz and Piccolo at the University of Maryland. Microsoft Research's Pivot (2010) and Deep Zoom brought the idea to mainstream tools. Prezi shipped a commercial ZUI for presentations in 2009.

The web caught up slowly. CSS transforms made smooth zoom cheap on every device, but nobody packaged the navigation model into a reusable engine — most "zoomable" demos were single-purpose. Zumly exists to change that: a small, framework-agnostic ZUI engine you can drop into any app, with a clean view contract and pluggable transition drivers.

Three principles that define a ZUI

1. Continuous spatial context

At any moment in a flat-page app, you see one screen. In a ZUI, you see the focused view and a hint of where it sits inside its parent. Even mid-transition, the previous view is still on the canvas (often dimmed and blurred) so the user understands the spatial relationship between where they were and where they're going.

2. The trigger is the destination

When you click a card to "open" it, the card itself zooms up to fill the viewport — it doesn't disappear and get replaced by a totally different page. This collapses two cognitive steps (which page am I leaving? which page am I arriving at?) into one spatial step.

3. Zoom out is the universal back

There is no "Back" button in the conventional sense. Zooming out reverses the trigger animation and returns you to the previous view, with all the surrounding context intact. Users learn this in seconds because it matches how zoom works everywhere else (maps, photos, microscopes).

Where ZUIs shine

  • Hierarchical data — dashboards, file systems, org charts. Anywhere the natural mental model is "drill into a subtree."
  • Spatially organised content — IoT and home control (rooms → devices → settings), maps and floor plans, factory and process diagrams.
  • Focus-heavy UIs — kiosks, signage, vehicle dashboards. Anywhere attention should be on one thing at a time but context matters.
  • Configurators and editors — product builders, design tools, CAD viewers — where users want to zoom into a part, edit it, and pop back without losing their place.

Where ZUIs are the wrong tool

ZUIs aren't a universal upgrade. They're a poor fit for:

  • Linear reading flows — long-form articles, blog posts, documentation sites. There's nothing to zoom into.
  • Forms and transactional flows — checkout, sign-up, settings pages. Users want a flat list of fields, not a spatial drill-down.
  • SEO-critical surfaces — every URL needs its own crawlable HTML page, and a ZUI built on a hash router collapses many states into one URL. Use static pages for the public surface and a ZUI for the authenticated app underneath.

How Zumly builds a ZUI

Zumly's job is the small, hard core of any ZUI: tracking which view is current, which is the parent, and which element triggered the zoom; computing the geometry that maps the trigger's bounding box to the canvas; and running a transition that animates from one to the other. You bring the views — Zumly makes them spatial.

Each view is a self-contained DOM subtree (or a mounted React/Vue/Svelte/Angular component). Triggers are any element you mark with class="zoom-me" and data-to="targetView". The engine handles depth tracking, lateral same-level navigation, and the transition driver (CSS, WAAPI, GSAP, Anime.js, Motion, or your own).
