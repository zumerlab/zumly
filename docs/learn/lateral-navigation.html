Lateral navigation

Lateral navigation moves you between sibling views — same parent, same depth, different content. It's the "swipe to the next photo," "next slide," "next room" primitive that complements Zumly's depth-based zooms.

Depth navigation → What is a ZUI?

Why lateral exists

A pure depth-only ZUI forces every move to go up or down the hierarchy. That works for true tree structures but it's miserable for the (very common) case of "I'm looking at sensor 7 in this room, and now I want to look at sensor 8 in the same room." With only zoom-in and zoom-out, the user has to back out to the parent and then drill into the sibling — three animations for what should be one step.

Lateral navigation collapses that into a single primitive: swap the current view for a sibling at the same depth, without touching the parent stack. The user sees one slide animation; the depth tracker doesn't move; back() still takes them out of the parent the way they expect.

How it differs from a zoom

A zoom is a spatial move: the trigger element grows or shrinks to fill the viewport. A lateral is a positional move: the current view slides off in one direction while a sibling slides in from the other. There's no scale change, no origin recalculation, no parent reflow — just two views translating across the canvas.

That's why Zumly's bundled drivers handle lateral transitions specially:

  • The default waapi driver runs a real slide animation with WAAPI keyframes.
  • The gsap, anime, and motion adapters delegate to runLateralInstant() — they don't add value to a slide that's already geometrically simple.
  • Custom drivers can call runLateralInstant(spec, onComplete) for the same instant behaviour, or write their own slide.

Where lateral shines

  • Photo galleries — next/previous photo at the same zoom level.
  • Carousels and slideshows — sibling slides under a single parent "deck" view.
  • Stepped flows — a multi-step wizard where each step is a sibling, not a child, and back() should exit the wizard rather than rewind one step.
  • Multi-tab focus — comparing sibling dashboards without losing the parent context.
  • Spatial neighbours — adjacent rooms in a building, adjacent stations on a production line, adjacent vehicles in a fleet.

Where lateral is the wrong call

Lateral isn't a substitute for depth. Reach for a zoom instead when:

  • The destination is inside the current view (a card you can see). That's a depth move — the trigger should grow into the new viewport.
  • The destination has a different parent. Zumly enforces same-depth-same-parent for lateral; cross-parent moves should round-trip through a shared ancestor.
  • Users would expect back() to undo the move. Lateral leaves the depth stack alone, so back() walks the parent stack — not the lateral history. If users say "wait, where did I come from?" you probably want a zoom-out + zoom-in instead.

Lateral and the depth stack

Because lateral doesn't change depth, the parent stack is untouched. If you laterally navigate from sibling A to sibling B at depth 3, the depth tracker still says "depth 3, parent at depth 2, grandparent at depth 1" — just with B in the current slot instead of A. Zoom-out from there takes you to depth 2, exactly where you'd land from A.

This is why lateral feels like the right tool for "sibling browsing" — it preserves the user's mental model of where they are in the hierarchy while still letting them move around at the current level.
