Zumly with Svelte

Add zoom-based navigation to any Svelte 5 application. Zumly is a framework-agnostic Zooming User Interface (ZUI) engine — your Svelte components mount into zoomable views using Svelte 5's mount() API, with no Zumly-aware code in your .svelte files.

Get started → View on GitHub

Why use Zumly with Svelte?

Svelte handles your component tree; Zumly handles the spatial relationship between trees. Each Zumly view is a self-contained DOM subtree where you call Svelte 5's mount(Component, { target, props }) exactly the same way you'd bootstrap a Svelte app into any other container element.

That means Zumly doesn't replace SvelteKit's router or compete with stores. It's an orthogonal layer for spatial, focus-driven navigation: instead of swapping URLs in a flat space, users zoom into a card, a row, an item, and the Svelte component for that view renders inside the zoomed-in target.

Common use cases for Zumly + Svelte:

  • Dashboards with hierarchical drill-down (KPIs → charts → raw data)
  • IoT and home control panels (rooms → devices → settings)
  • Settings UIs that zoom into nested sections instead of pushing routes
  • Kiosk and signage interfaces where focus is the primary navigation metaphor
  • Product configurators with zoomable detail levels

Install

npm install zumly

Zumly has zero dependencies. The package ships ESM, CJS, and a UMD build for plain HTML.

Quick start

The simplest pattern: declare your views as function views. Zumly hands you a fresh target element, and you mount a Svelte component into it with Svelte 5's mount() function. 

import Zumly from 'zumly'
import 'zumly/css'
import { mount } from 'svelte'
import Dashboard from './Dashboard.svelte'

const app = new Zumly({
  mount: '.canvas',
  initialView: 'home',
  views: {
    home: '<div class="z-view"><div class="zoom-me" data-to="dashboard" data-id="42">Open dashboard</div></div>',
    dashboard: ({ target, props }) => {
      mount(Dashboard, { target, props: { id: props.id } })
    }
  }
})

await app.init()

Any element inside a view with class zoom-me and a data-to attribute becomes a zoom trigger. Data attributes on the trigger (like data-id="42") get passed into your Svelte component as props.id.

Svelte 4? Replace mount(Dashboard, { target, props }) with new Dashboard({ target, props }). The rest of the integration is identical.

Working example: Dashboard with drill-down

A two-level pattern where a list of items zooms into a detail view, with the Svelte component receiving the selected item's id from the trigger.

App.svelte

<script>
  import { onMount, onDestroy } from 'svelte'
  import { mount } from 'svelte'
  import Zumly from 'zumly'
  import 'zumly/css'

  import ItemList from './ItemList.svelte'
  import ItemDetail from './ItemDetail.svelte'

  let canvas
  let zumly

  onMount(async () => {
    zumly = new Zumly({
      mount: canvas,
      initialView: 'list',
      views: {
        list: ({ target }) => {
          mount(ItemList, { target })
        },
        detail: ({ target, props }) => {
          mount(ItemDetail, { target, props: { id: props.id } })
        }
      },
      transitions: {
        duration: '600ms',
        ease: 'cubic-bezier(.4,0,.2,1)',
        effects: ['blur(4px) brightness(0.6)', 'blur(10px) brightness(0.3)']
      }
    })
    await zumly.init()
  })

  onDestroy(() => zumly?.destroy())
</script>

<div bind:this={canvas} class="zumly-canvas" style="height: 100vh"></div>

ItemList.svelte

<script>
  const items = [
    { id: '1', name: 'Engine telemetry' },
    { id: '2', name: 'Battery health' },
    { id: '3', name: 'Brake system' }
  ]
</script>

<div class="z-view">
  <h1>Subsystems</h1>
  {#each items as item (item.id)}
    <div class="zoom-me" data-to="detail" data-id={item.id}>
      {item.name}
    </div>
  {/each}
</div>

ItemDetail.svelte (Svelte 5)

<script>
  let { id } = $props()
</script>

<div class="z-view">
  <h1>Detail for {id}</h1>
</div>

Lifecycle and cleanup

Function views in Zumly are never cached — they resolve fresh each time the user zooms in. That's the right behavior for Svelte: every zoom-in mounts a new component instance, every zoom-out unmounts it, and Svelte's lifecycle hooks (onMount, onDestroy, etc.) run as expected.

Tip: If you need explicit teardown of a Svelte 5 component (for example, to release a long-lived resource), capture the instance returned by mount() and call unmount(instance) from a Zumly 'beforeZoomOut' event. For most components, Zumly's standard cleanup is enough.

Sharing context (stores, runes, services)

Pass shared state — Svelte stores, rune-based state objects, API clients — to every view via componentContext on the Zumly constructor. Zumly forwards it as context to all function and object views, so each Svelte component can read from a single source of truth. 

import { writable } from 'svelte/store'

const userStore = writable({ id: 1, name: 'Ada' })

const app = new Zumly({
  mount: '.canvas',
  initialView: 'home',
  componentContext: { userStore },
  views: {
    dashboard: ({ target, context }) => {
      mount(Dashboard, {
        target,
        props: { user: context.userStore }
      })
    }
  }
})

TypeScript

Zumly ships TypeScript types out of the box. View functions are typed with the target, props, and context parameters, so your Svelte mount code gets full IntelliSense. 

import Zumly, { ViewFunction } from 'zumly'
import { mount } from 'svelte'
import Dashboard from './Dashboard.svelte'

const dashboard: ViewFunction = ({ target, props }) => {
  mount(Dashboard, { target, props: { id: props.id as string } })
}

Common gotchas

  • Use Svelte 5's mount(), not new Component(). Svelte 5 deprecated the constructor pattern. The new API takes (Component, { target, props }).
  • The target is a fresh <div> created by Zumly — don't try to mount into document.body or query for an existing element. Just call mount(Component, { target }).
  • Don't forget the z-view class on your Svelte component's root element. Zumly auto-adds it if missing, but being explicit avoids surprises with CSS selectors. Note that Svelte may wrap your component in an extra parent element — Zumly handles that in its cleanup logic.
  • Trigger data attributes become strings. data-id="42" arrives as props.id === "42", not as a number. Cast or parse on the Svelte side.
