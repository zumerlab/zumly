Zumly with Angular

Add zoom-based navigation to any Angular application. Zumly is a framework-agnostic Zooming User Interface (ZUI) engine — your standalone Angular components mount into zoomable views via createApplication and bootstrap(), with no Zumly-aware code in your components.

Get started → View on GitHub

Why use Zumly with Angular?

Angular handles your component tree; Zumly handles the spatial relationship between trees. Each Zumly view is a self-contained DOM subtree where you bootstrap a standalone Angular component using createApplication and the application's bootstrap() method, exactly the same way you'd bootstrap Angular into any other root element.

That means Zumly doesn't replace the Angular Router or compete with services. It's an orthogonal layer for spatial, focus-driven navigation: instead of swapping URLs in a flat space, users zoom into a card, a row, an item, and the Angular component for that view renders inside the zoomed-in target.

Common use cases for Zumly + Angular:

  • Enterprise dashboards with hierarchical drill-down (KPIs → charts → raw data)
  • Industrial control panels (lines → stations → sensors)
  • Settings UIs that zoom into nested sections instead of pushing routes
  • Kiosk and signage interfaces where focus is the primary navigation metaphor
  • Configurators with zoomable detail levels

Install

npm install zumly

Zumly has zero dependencies. The package ships ESM, CJS, and a UMD build for plain HTML.

Quick start

The cleanest pattern uses Angular's standalone API: declare your views as function views, and inside each one, call createApplication to spin up an Angular environment, then bootstrap() a standalone component into the target element Zumly hands you. 

import Zumly from 'zumly'
import 'zumly/css'
import { createApplication } from '@angular/platform-browser'
import { DashboardComponent } from './dashboard.component'

const app = new Zumly({
  mount: '.canvas',
  initialView: 'home',
  views: {
    home: '<div class="z-view"><div class="zoom-me" data-to="dashboard" data-id="42">Open dashboard</div></div>',
    dashboard: async ({ target, props }) => {
      const ngApp = await createApplication({})
      const ref = ngApp.bootstrap(DashboardComponent, target)
      ref.setInput('id', props.id)
    }
  }
})

await app.init()

Any element inside a view with class zoom-me and a data-to attribute becomes a zoom trigger. Data attributes on the trigger (like data-id="42") get passed into your Angular component as props.id, which you forward via setInput().

Standalone components required. This guide uses Angular's standalone component API (Angular 14+). If you're still on NgModules, wrap your component in a bootstrappable module first.

Working example: Dashboard with drill-down

A two-level pattern where a list of items zooms into a detail view, with the Angular component receiving the selected item's id from the trigger as a typed @Input.

app.component.ts

import { Component, ElementRef, AfterViewInit, OnDestroy, ViewChild } from '@angular/core'
import { createApplication } from '@angular/platform-browser'
import Zumly from 'zumly'
import 'zumly/css'

import { ItemListComponent } from './item-list.component'
import { ItemDetailComponent } from './item-detail.component'

@Component({
  selector: 'app-root',
  standalone: true,
  template: `<div #canvas class="zumly-canvas" style="height: 100vh"></div>`
})
export class AppComponent implements AfterViewInit, OnDestroy {
  @ViewChild('canvas', { static: true }) canvas!: ElementRef<HTMLDivElement>
  private zumly: any

  async ngAfterViewInit() {
    this.zumly = new Zumly({
      mount: this.canvas.nativeElement,
      initialView: 'list',
      views: {
        list: async ({ target }) => {
          const app = await createApplication({})
          app.bootstrap(ItemListComponent, target)
        },
        detail: async ({ target, props }) => {
          const app = await createApplication({})
          const ref = app.bootstrap(ItemDetailComponent, target)
          ref.setInput('id', props.id)
        }
      },
      transitions: {
        duration: '600ms',
        ease: 'cubic-bezier(.4,0,.2,1)',
        effects: ['blur(4px) brightness(0.6)', 'blur(10px) brightness(0.3)']
      }
    })
    await this.zumly.init()
  }

  ngOnDestroy() {
    this.zumly?.destroy()
  }
}

item-list.component.ts

import { Component } from '@angular/core'

@Component({
  selector: 'item-list',
  standalone: true,
  template: `
    <div class="z-view">
      <h1>Subsystems</h1>
      <div
        *ngFor="let item of items"
        class="zoom-me"
        data-to="detail"
        [attr.data-id]="item.id">
        {{ item.name }}
      </div>
    </div>
  `
})
export class ItemListComponent {
  items = [
    { id: '1', name: 'Engine telemetry' },
    { id: '2', name: 'Battery health' },
    { id: '3', name: 'Brake system' }
  ]
}

item-detail.component.ts

import { Component, Input } from '@angular/core'

@Component({
  selector: 'item-detail',
  standalone: true,
  template: `
    <div class="z-view">
      <h1>Detail for {{ id }}</h1>
    </div>
  `
})
export class ItemDetailComponent {
  @Input() id?: string
}

Lifecycle and cleanup

Function views in Zumly are never cached — they resolve fresh each time the user zooms in. Each zoom-in creates a new Angular ApplicationRef via createApplication, and Zumly's standard cleanup tears down the DOM on zoom-out. Angular's lifecycle hooks (ngOnInit, ngOnDestroy, etc.) run as expected within each view.

Tip: Each createApplication call has a non-trivial cost (it spins up a fresh Angular zone and DI tree). For high-frequency navigation, prefer an object view with render() and mounted(), and keep the ApplicationRef in a closure so it can be reused across mounts.

Sharing context (services, providers, DI)

Pass shared providers — services, HTTP clients, route guards — via the providers array on createApplication. For Zumly-managed shared state, you can also forward objects via componentContext, which Zumly hands to every view as context. 

import { provideHttpClient } from '@angular/common/http'
import { UserService } from './user.service'

const app = new Zumly({
  mount: '.canvas',
  initialView: 'home',
  views: {
    dashboard: async ({ target, props }) => {
      const ngApp = await createApplication({
        providers: [
          provideHttpClient(),
          UserService
        ]
      })
      const ref = ngApp.bootstrap(DashboardComponent, target)
      ref.setInput('userId', props.id)
    }
  }
})

TypeScript

Zumly ships TypeScript types out of the box — and Angular is TypeScript-first by design, so the integration is fully typed end to end. View functions are typed with the target, props, and context parameters, and component @Inputs give you compile-time safety on the values you forward from triggers. 

import Zumly, { ViewFunction } from 'zumly'
import { createApplication } from '@angular/platform-browser'
import { DashboardComponent } from './dashboard.component'

const dashboard: ViewFunction = async ({ target, props }) => {
  const app = await createApplication({})
  const ref = app.bootstrap(DashboardComponent, target)
  ref.setInput('id', props.id as string)
}

Common gotchas

  • Use standalone components. The integration assumes Angular 14+ standalone components. NgModule-based apps need an extra wrapping step that's not covered here.
  • The target is a fresh <div> created by Zumly — don't try to bootstrap into document.body. Pass the target directly as the second argument to bootstrap().
  • Use setInput(), not direct property assignment, when passing data from props to your component. setInput() triggers Angular's change detection correctly.
  • Trigger data attributes become strings. data-id="42" arrives as props.id === "42", not as a number. Cast or parse on the Angular side, and type your @Input accordingly.
  • Don't forget the z-view class on your component's template root. Zumly auto-adds it if missing, but being explicit avoids surprises with CSS selectors.
