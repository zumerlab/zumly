Zumly with React

Add zoom-based navigation to any React application. Zumly is a framework-agnostic Zooming User Interface (ZUI) engine — your React components mount into zoomable views with no wrapper, no bridge library, and no Zumly-aware code in your component tree.

Get started → View on GitHub

Why use Zumly with React?

React handles your component tree; Zumly handles the spatial relationship between trees. Each Zumly view is a self-contained DOM subtree where you can mount a React root, render a component, and tear it down on zoom-out — exactly the same way you'd mount React into any other container element.

That means Zumly doesn't replace React Router or compete with your state management. It's an orthogonal layer for spatial, focus-driven navigation: instead of swapping URLs in a flat space, users zoom into a card, a row, an item, and the React component for that view renders inside the zoomed-in target.

Common use cases for Zumly + React:

  • Dashboards with hierarchical drill-down (KPIs → charts → raw data)
  • IoT and home control panels (rooms → devices → settings)
  • Settings UIs that zoom into nested sections instead of pushing pages
  • Kiosk and signage interfaces where focus is the primary navigation metaphor
  • Product configurators with zoomable detail levels

Install

npm install zumly

Zumly has zero dependencies. The package ships ESM, CJS, and a UMD build for plain HTML.

Quick start

The simplest pattern: declare your views as function views. Zumly hands you a fresh target element, and you mount a React root into it with createRoot. 

import Zumly from 'zumly'
import 'zumly/css'
import { createRoot } from 'react-dom/client'
import Dashboard from './Dashboard'

const app = new Zumly({
  mount: '.canvas',
  initialView: 'home',
  views: {
    home: '<div class="z-view"><div class="zoom-me" data-to="dashboard" data-id="42">Open dashboard</div></div>',
    dashboard: ({ target, props }) => {
      const root = createRoot(target)
      root.render(<Dashboard id={props.id} />)
    }
  }
})

await app.init()

That's it. Any element inside a view with class zoom-me and a data-to attribute becomes a zoom trigger. Data attributes on the trigger (like data-id="42") get passed into your React component as props.id.

Working example: Dashboard with drill-down

A two-level pattern where a list of items zooms into a detail view, with the React component receiving the selected item's id from the trigger. 

// App.jsx
import { useEffect, useRef } from 'react'
import Zumly from 'zumly'
import { createRoot } from 'react-dom/client'
import 'zumly/css'

import ItemList from './ItemList'
import ItemDetail from './ItemDetail'

export default function App() {
  const canvasRef = useRef(null)

  useEffect(() => {
    const app = new Zumly({
      mount: canvasRef.current,
      initialView: 'list',
      views: {
        list: ({ target }) => {
          createRoot(target).render(<ItemList />)
        },
        detail: ({ target, props }) => {
          createRoot(target).render(<ItemDetail id={props.id} />)
        }
      },
      transitions: {
        duration: '600ms',
        ease: 'cubic-bezier(.4,0,.2,1)',
        effects: ['blur(4px) brightness(0.6)', 'blur(10px) brightness(0.3)']
      }
    })
    app.init()
    return () => app.destroy()
  }, [])

  return <div ref={canvasRef} className="zumly-canvas" style={{ height: '100vh' }} />
}
// ItemList.jsx
export default function ItemList() {
  const items = [
    { id: '1', name: 'Engine telemetry' },
    { id: '2', name: 'Battery health' },
    { id: '3', name: 'Brake system' }
  ]
  return (
    <div className="z-view">
      <h1>Subsystems</h1>
      {items.map(item => (
        <div key={item.id} className="zoom-me" data-to="detail" data-id={item.id}>
          {item.name}
        </div>
      ))}
    </div>
  )
}

Lifecycle and cleanup

Function views in Zumly are never cached — they resolve fresh each time the user zooms in. That's the right behavior for React: every zoom-in mounts a new React root, every zoom-out unmounts it, and React's normal lifecycle hooks (useEffect cleanup, useLayoutEffect, etc.) work as expected.

Tip: If you need to keep a component instance alive across navigation (for example, a video player that shouldn't re-mount), use an object view with render() and mounted() methods, and store your React root in a closure variable instead of recreating it.

Sharing context (Redux, Zustand, React Context)

Pass your store or context provider to every view via componentContext on the constructor. Zumly forwards it as context to all function and object views, so each React subtree can wrap its rendered tree in the same provider. 

const store = createStore(...)

const app = new Zumly({
  mount: '.canvas',
  initialView: 'home',
  componentContext: { store },
  views: {
    dashboard: ({ target, context }) => {
      createRoot(target).render(
        <Provider store={context.store}>
          <Dashboard />
        </Provider>
      )
    }
  }
})

TypeScript

Zumly ships TypeScript types out of the box. View functions are typed with the target, props, and context parameters, so your React mount code gets full IntelliSense. 

import Zumly, { ViewFunction } from 'zumly'

const dashboard: ViewFunction = ({ target, props }) => {
  createRoot(target).render(<Dashboard id={props.id as string} />)
}

Common gotchas

  • The target is a fresh <div> created by Zumly — don't try to mount into document.body or query for an existing element. Just call createRoot(target).render(...).
  • Don't forget the z-view class on your component's root element if you're returning JSX with a wrapper. Zumly auto-adds it if missing, but being explicit avoids surprises with CSS selectors.
  • Trigger data attributes become strings. data-id="42" arrives as props.id === "42", not as a number. Cast or parse on the React side.
  • Style isolation: each view is a normal DOM subtree, not a Shadow DOM. Your global React styles apply normally — but so do parent view styles, so scope carefully if you mix CSS-in-JS and global stylesheets.
